## 2부. 핵심 질문 — "이게 게임을 재밌게 만드는가?"

### 평가 프레임: 재미의 3원칙
1. **가시성(Legibility)** — 플레이어가 원인→결과를 *볼 수 있어야* 시스템이 존재하는 의미가 있다.
2. **결정(Decisions)** — 좋은 시스템은 플레이어에게 *새로운 선택*을 만들어준다. 결정 없는 시뮬레이션은 배경 소음이다.
3. **드라마(Drama)** — 이야기는 갈등에서 나온다. 긍정 피드백만 있는 시스템은 이야기가 없다.

### 판정: **방향은 맞다. 그러나 현재 구현은 3원칙을 전부 비켜갔다.**

| 결함 | 내용 | 위반 원칙 |
|---|---|---|
| **🔴 D1. 무드가 완전히 비가시적** | 무드 UI가 0이다 — 크루 카드에 칩 없음, 툴팁 없음, 무드 하락 보고서 없음. ±12% 작업 속도 변화는 "훈련이 360분 대신 400분 걸림" 수준이라 플레이어가 인지 자체를 못 한다. **보이지 않는 배율 = 복잡도 비용만 내고 재미는 0.** | 가시성 |
| **🔴 D2. 관계가 단방향 랫칫(ratchet)** | `updateRelationshipsFromActivities`는 친밀도를 **올리기만 한다**(+1~2/틱). 감소 경로가 전혀 없다 — 갈등도, 성격 충돌도, 소원해짐(decay)도 없음. 결과: friction 밴드(≤-35)는 **실플레이에서 도달 불가능한 죽은 코드**이고, 전 승무원이 close로 수렴해 "항구적 +4% 버프"라는 밋밋한 보너스로 끝난다. RimWorld 관계가 재밌는 이유는 싸움·연애·결별이지, 전원 친구가 아니다. | 드라마 |
| **🟠 D3. 수렴 속도 과속** | crewAI 틱(12게임분)마다 +1~2 → 같은 방에서 **게임 내 4~7시간이면 close 도달**, 이후 상한 100까지 계속 누적. 관계가 "형성되는 과정"이 존재하지 않는다. 또한 영입으로 승무원이 늘면 쌍이 N²로 늘고 죽은 승무원 쌍도 영구 보존(정리 없음). | 드라마·기술 |
| **🟠 D4. 특성이 영원한 장식** | 21-A가 표시 전용인 것 자체는 옳은 첫 수지만, 로드맵(21-D)에 특성→무드/관계/작업 **연결 계획이 없다**. 특성의 재미 = 승무원마다 *다른 이야기*가 생기는 것. 연결 없는 특성 칩은 첫 열람 후 아무 의미가 없다. | 결정·드라마 |
| **🟡 D5. 플레이어 결정이 하나도 안 늘었다** | 무드/관계가 생겨도 플레이어가 *할 수 있는 것*이 없다 — 방 배치 지정 불가, 휴식 정책의 무드 연동 없음, 관계 관리 수단 없음. 현재 형태는 플레이어와 무관하게 돌아가는 내부 수치다. | 결정 |
| **🟡 P1. 프로세스 위반** | PR 제목은 "21-A"인데 실제 A+B+C 3슬라이스가 한 PR에 — 작은 PR 규칙(PROJECT_GUIDE §9) 위반. 테스트는 통과하나 리뷰 단위가 커졌다. | — |

### 한 줄 결론
> Codex는 **시뮬레이션의 뼈대**를 규칙대로 잘 깔았다. 하지만 이대로 머지하고 끝내면
> "플레이어 몰래 돌아가는 수치 노이즈"가 하나 늘어날 뿐이다. **뼈대는 살리고,
> 가시성·드라마·결정을 붙이는 후속 슬라이스가 반드시 뒤따라야 한다.**

## 3부. 수정 계획 (실행 지시서)

### PR #98 처리 방안 — **조건부 수용**
- 테스트 435 통과 + 아키텍처 준수 + 뼈대 재사용 가치 → **머지한다.**
- 단, 머지는 아래 21-D′~F′ 착수의 전제이지 완결이 아니다. D1·D2는 방치 시 실질적 디자인 버그이므로
  **머지 직후 21-D′(가시화)와 21-E′(드라마)를 연속 진행**한다.
- 머지 전 확인: master(버그 라운드 21) 대비 충돌 여부, CI 통과.

### 후속 슬라이스 (기존 21-D 재설계 → D′~F′)

#### 21-D′ — 무드·관계 가시화 (D1·D5 해소, 소형)
목표: 이미 돌아가는 수치를 플레이어 눈에 보이게. **새 게임 수치 변경 없음.**
- 크루 카드: 무드 칩(밴드 라벨+점수, `deriveCrewMood` 재사용) + 무드 구성 요인 툴팁("스트레스 높음 · 수면 부족").
- 크루 카드/상세: 관계 요약(가장 친한 동료 1명, 불화 동료 1명 — friction이 생기는 21-E′ 이후 의미).
- ShipInterior 마커: 저하(low) 무드 승무원 시각 표시(기존 우선순위 점 패턴 재사용, 과하면 생략 판단 허용).
- 보고서 연동: 무드 밴드가 **저하(low)로 떨어질 때 1회** crew 카테고리 보고서(쓰로틀 필수 — 19-B Map 패턴).
  회복 시 보고서 없음(볼륨 선별 원칙).

#### 21-E′ — 관계에 드라마 주입 (D2·D3 해소, 핵심)
목표: 단방향 랫칫을 양방향 동역학으로. **여기가 Phase 21의 성패를 가른다.**
- **특성 궁합**: 특성 쌍별 호환 보정 — 예: 승부욕↔승부욕 = 마찰 성향, 원칙주의↔승부욕 = 마찰 성향,
  보살핌↔누구나 = 친화 성향, 침착함 = 중립 안정화. 궁합이 나쁜 쌍은 같은 방 근무 시 친밀도가 **깎인다**
  (D4 해소의 첫 연결 — 특성이 처음으로 기계적 의미를 가짐).
- **언쟁(spat) 이벤트**: 마찰 쌍이 같은 방 + 둘 중 하나가 긴장/저하 무드일 때 낮은 확률로 언쟁 발생 —
  친밀도 급락, 양쪽 스트레스 상승, **crisis 아닌 crew 보고서 발행**("강로운과 민수리 언쟁 — 같은 방 배치 재고").
- **소원해짐(decay)**: 함께 근무하지 않는 쌍은 친밀도가 0을 향해 서서히 회귀(관계는 유지 노력이 필요).
- **속도 재조정**: 축적 속도를 현행의 1/4~1/6 수준으로(수 게임일에 걸쳐 형성), close 상한 도달을 이벤트화.
- **정리(pruning)**: 사망 승무원 쌍은 merge/업데이트 시 제거.
- 수치는 전부 `data/` 카탈로그로(궁합표·확률·감쇠율) — 밸런싱 가능하게.

#### 21-F′ — 특성 기계화: "특성당 명확한 규칙 1개" (D4 해소)
목표: 각 특성이 **플레이어가 문장으로 설명 가능한** 딱 하나의 효과를 가진다(툴팁에 명시).
- 침착함: 위기 대응 속도 보정 / 승부욕: 전투 전술 보정 + 전투 없는 장기간에 무드 하락 압력 /
  보살핌: 치료·회복 잡 속도 보정 / 기계친화: 기관실 방작업 보정 / 호기심: 조우 "조사" 계열 선택지 보상 보정 /
  원칙주의: 정책 자동화 대상일 때 효율 보정. *(정확한 대상·수치는 착수 시 카탈로그로 확정, 효과는 소폭 ±10~15%)*
- 영입(가챠) 후보 카드에 특성 노출 → **영입이 스탯 뽑기에서 "사람 뽑기"로** — 결정의 질이 달라지는 지점.
- crewAI 우선순위 스택(치료>위기>강제휴식>큐>방작업)은 **불변**. 효과는 전부 선택 이후의 배율/보정.

#### 21-G′ — 안정화 + 문서 (기존 F 상당)
- 4정책 ON + 관계 동역학 포함 장기(100틱+) 통합 테스트: 친밀도 분포가 전원-close로 붕괴하지 않는지,
  언쟁 보고서 볼륨 상한, 세이브 왕복.
- `PHASE_21_INNER_LIFE.md` 최종화(이 보고서의 결함→해소 매핑 포함), PROJECT_GUIDE §6·7 갱신.

### 실행 순서·프로세스
```
PR #98 머지 (조건부 수용) → 21-D′ (가시화, 1 PR) → 21-E′ (드라마, 1~2 PR) → 21-F′ (특성 기계화, 1~2 PR) → 21-G′ (안정화)
```
- 각 PR: 작은 단위 · 신규 테스트 · `npm test`/`npm run build` 통과 · Playwright 검증 · CI 후 squash 머지 (PROJECT_GUIDE §9).
- 구현은 하위 에이전트 위임, 감독 세션이 스펙·검수·머지 담당(기존 체계 유지).
- 각 슬라이스 스펙에 이 문서의 결함 ID(D1~D5)를 인용해 "무엇을 해소하는 슬라이스인지" 명시할 것.

### 성공 판정 기준 (Phase 21이 "재밌어졌다"의 정의)
1. 플레이어가 크루 화면만 보고 "누가 왜 컨디션이 나쁜지" 말할 수 있다. (가시성)
2. 보고서 수신함에 승무원 '사람 이야기'(언쟁·단짝·무드 급락)가 흘러들어온다. (드라마)
3. 영입·방 배치·휴식에서 특성/관계 때문에 고민되는 순간이 생긴다. (결정)
4. 테스트 전부 통과 + 세이브 호환 + crewAI 우선순위 불변. (안정)
